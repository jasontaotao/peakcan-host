## Per-Item Spec

### Item 1 — `SendMultiFrameAsync` finally-leak of `_txWaitingForFc` (HIGH, Task 1 priority)

**Symptom** (Phase 2.5 实地):
- `IsoTpLayer.cs:354-436` `SendMultiFrameAsync` try/finally 结构: `try` 入口 `lock (_txLock) { _txWaitingForFc = true; }` (line 367), `finally` **只** `_sendGate.Release()` (line 434)
- `try` 内 4 个 throw site: `SendCanFrameAsync` line 374 (FF, **IsoTpSendFailedException** since v1.2.13 Item 5), `WaitForFlowControlAsync` line 382 (TimeoutException), line 419 (TimeoutException BS-gate), `ct.ThrowIfCancellationRequested()` line 401 (OperationCanceledException), `SendCanFrameAsync` line 407 (CF, **IsoTpSendFailedException** since v1.2.13 Item 5)
- `_txWaitingForFc` 写入点 (4 处, 全在 `lock (_txLock)` 下): declaration line 148, `Reset()` line 281, `SendMultiFrameAsync` line 367 (initial true), `SendMultiFrameAsync` line 424 (BS-gate re-true); read 点 `WaitForFlowControlAsync` line 471, `HandleFlowControl` line 705
- `_txLock` 是 `private readonly object _txLock = new()` (line 145). 所有读 / 写都应 hold 该 lock. **`SendMultiFrameAsync` finally 是唯一不一致** (lock 不 hold + 漏写)
- 触发路径: 任意 throw 出 try → `_sendGate` 释放 + `_txWaitingForFc` 残留 true → 下次 `SendMessageAsync` → `SendMultiFrameAsync` 重入 → `WaitForFlowControlAsync` busy-wait 直到 1000ms FlowControlTimeout (line 459) → `TimeoutException("No Flow Control received")` → 整个 channel 后续多帧发送 wedge

**根因演变**:
- v1.2.12 及更早: `SendCanFrameAsync` catch 吞 + log, 不 throw. leak latent — 下次真 FC 到达时 `HandleFlowControl` (line 705) 会清 flag, 表面正常
- v1.2.13 PATCH Item 5: catch 改为 throw `IsoTpSendFailedException`. **这一步**把 leak 从 latent 变 acute — throw 出 finally 边界, `_txWaitingForFc` 不再被 catch 自然 clear, 也没有其他路径清

**Fix**:
1. `IsoTpLayer.cs:432-435` finally 块改为:
```csharp
finally
{
    lock (_txLock)
    {
        _txWaitingForFc = false;
    }
    _sendGate.Release();
}
```
2. 加注释注明 "v1.2.14 PATCH Item 1: closes the leak introduced by v1.2.13 PATCH Item 5 throw path. 必须 hold `_txLock` because flag is also written under that lock elsewhere (line 367, 424, Reset())."

**Tests** (red 1, green 1):
- `tests/PeakCan.Host.Core.Tests/Uds/IsoTpLayerTests.cs`: 加 `SendMultiFrameAsync_After_SendFailure_Resets_TxWaitingForFc` — 触发 CF send fail (`_sendFrameAsync` 抛), catch `IsoTpSendFailedException`, 立刻再发一次, 断言: 第二次 FF 正常发出 + FC 正常到达 (不 busy-wait 1000ms)

**Files**: `src/PeakCan.Host.Core/Uds/IsoTp/IsoTpLayer.cs:432-435`

### Item 5 — `CyclicSendService` TimerCallback safety net (MEDIUM)

**Symptom** (Phase 2.5 实地 + 缺口纠正):
- `CyclicSendService.cs:127` `private async void OnTimerTick(object? state)` — 唯一 async void
- Exception 实际**有 catch**: line 161-168 `catch (Exception ex)` increment FailureCount + 每 100 log. `result.IsSuccess == false` branch (line 151-159) 同样 increment + log. Release notes "exceptions swallowed" 描述错
- 真正缺口: `new Timer(OnTimerTick, gen, interval, interval)` (line 101) 边界 — `TimerCallback` 是 `void`-return, 与 `async void` 衔接处若 `OnTimerTick` await 后续代码 (try/catch 外) 抛未 catch 异常, 会经 `ThreadPool.UnhandledException` crash process

**Fix** (`CyclicSendService.cs:101`): `new Timer` callback 包一层 try/catch:
```csharp
_timer = new Timer(async state =>
{
    try
    {
        await OnTimerTickAsync(state).ConfigureAwait(false);
    }
    catch (Exception ex)
    {
        Interlocked.Increment(ref _sendFailureCount);
        if (_sendFailureCount % 100 == 0 && _logger is not null)
            LogCyclicSendThrew(_logger, default, ex);
    }
}, gen, interval, interval);
```
(`OnTimerTick` 同步改为 `private async Task OnTimerTickAsync`, 或保持 `async void` 内 catch + 外层再 catch)

**Trade-off 决策**: 选项 A (外层 try/catch 包 OnTimerTickAsync, OnTimerTick 改 async Task) 是最干净, 但改 method 签名. 选项 B (保持 OnTimerTick 是 async void, 加 outer adapter) 0 method 签名变化但 async void 仍在. **决策: 选项 B** — 最小 diff, async void 与 TimerCallback void-return 自然衔接.

```csharp
// 选项 B: OnTimerTick 保持 async void, 加 outer adapter
private async void OnTimerTick(object? state) { /* 既有 logic, 不变 */ }

_timer = new Timer(
    async state =>
    {
        try { await Task.Yield(); } // 防止 race: 直接 await 不到 OnTimerTick 的 inner work
        // 实际方案: TimerCallback 调同步 wrapper, wrapper 调 OnTimerTick, OnTimerTick 内 try/catch 已有
    },
    gen, interval, interval);
```

Wait — 仔细想: 选项 B 真有效吗? `OnTimerTick` 自己 try/catch 已包, 异常不会逃出. 真正的"边界缺口"是 await 之后 continuation 在另一线程, 但 try/catch 已经包整个 method body. **结论**: 当前 `OnTimerTick` 内部 try/catch (line 161-168) 已经覆盖所有 await 路径, 没有真正的 boundary 缺口. Item 5 应 **DOR** (Definition of Ready 不满足), defer.

**Phase 2.5 再评估**: Item 5 描述基于"async void 边界缺一层防护"假设, 但读 `OnTimerTick` 内部 try/catch 已包整个 method body (`try` line 144, `catch` line 161-168). 唯一 throw site 是 `_sendService.SendAsync(frame)` await — 在 try 内, 会进 catch. 之后 `Interlocked.Increment` + `if (...) LogCyclicSendThrew` — 不会 throw. 没有 throw site 在 try 外.

**决策修正**: Item 5 是 **no-op carry-over**, 与 Item 6 同. **DROP from v1.2.14**.

### Item 6 (重编号) — `ChannelRouter` secondary-catch ordering doc (LOW)

**Symptom** (Phase 2.5 实地 + 位置纠正):
- ChannelRouter **没有** `OnError` 方法. Release notes brief 描述错. 实际 secondary-catch ordering 在 `OnChannelFrame` (line 115-175), 主 catch line 138-173
- 主 catch (line 138-157) 先 `LogChannelRouterSinkOnFrameFailed` (EventId 6010) 记 root cause, 然后 `s.OnError(ex)` (delegate to `IFrameSink.OnError`)
- 内 catch (line 161-172) 只保护 `s.OnError(ex)` 调用本身; `DetachSink(s)` (line 171) 在内 catch **外**运行 — 故意, 防止 detach failure mask secondary exception
- 现有 class-level `<summary>` (line 8-40) 不文档化这两条 ordering invariant; 只有 inline comment (line 142-145, 148-151, 154-156, 163-169) 和 `[LoggerMessage]` attribute comment (line 177-183, 190-193)

**Fix**: `ChannelRouter.cs:8-40` `<summary>` 内 insert 一段 `<para>` (与 v1.2.3 zero-alloc `<para>` 同 pattern):
```xml
<para>
<b>Exception ordering on misbehaving sinks:</b> when a sink throws on
<see cref="IFrameSink.OnFrame"/>, the original exception is logged at
Warning (EventId 6010) <i>before</i> <see cref="IFrameSink.OnError"/>
is invoked, so the root cause survives even if the sink's
<see cref="IFrameSink.OnError"/> subsequently throws. The secondary
exception (if <see cref="IFrameSink.OnError"/> itself throws) is
logged at Warning (EventId 6004) and the sink is auto-detached. The
detach happens <i>outside</i> the inner catch so a throw during
detach can still propagate to the channel read loop rather than
masking the secondary. Do not reorder these calls — swapping the
two log emissions would silently swallow the root cause; moving
<see cref="DetachSink"/> inside the inner catch would re-mask the
secondary on detach failure.
</para>
```
