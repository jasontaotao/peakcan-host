## Item 1 ‚Äî Trace `FrameType` distinguishes RTR

### Symptom

`TraceEntry.FrameType` (`src/PeakCan.Host.App/ViewModels/TraceEntry.cs:47-48`) returns `""` for both Standard CAN and RTR frames ‚Äî they look identical in the Type column.

```csharp
public string FrameType => IsError ? "ERR" : IsFd ? "FD" : "";
```

`CanFrame.Flags` carries `FrameFlags.Rtr` (`src/PeakCan.Host.Core/FrameFlags.cs:13`) but `TraceEntry` does not project it. `DbcService` lookup for RTR frames is by design skipped (RTR carries no payload to decode), but the trace row should still show "RTR" in the Type column for the user's visual distinction.

### Fix

1. Add `IsRtr` derived property to `TraceEntry`:
   ```csharp
   public bool IsRtr { get; init; }   // init-only is fine (set once at construction)
   ```
2. Extend `FrameType`:
   ```csharp
   public string FrameType => IsError ? "ERR" : IsRtr ? "RTR" : IsFd ? "FD" : "";
   ```
   **ERR takes precedence over RTR** (matches existing "ERR before FD" priority; both can technically co-occur).
3. `TraceViewModel.AppendBatchAsync` already maps `f.IsError` / `f.IsFd`; add `IsRtr = f.Flags has FrameFlags.Rtr` to the `new TraceEntry` initializer.

### Behavior matrix

| Frame kind | IsError | IsRtr | IsFd | FrameType | Row background |
|------------|---------|-------|------|-----------|----------------|
| Standard CAN | false | false | false | `""` | default |
| RTR | false | true | false | `"RTR"` | default |
| CAN FD | false | false | true | `"FD"` | `#E3F2FD` (light blue) |
| CAN FD + BRS | false | false | true (BRS) | `"FD"` | `#E3F2FD` |
| Error | true | any | any | `"ERR"` | `#FFCDD2` (light red) |

(RTR row keeps default background ‚Äî color contrast against the red/blue is enough.)

- `TraceEntryTests.FrameType_Rtr_With_Fd_Still_Returns_Rtr` (RTR + FD is valid in CAN FD)

## Item 2 ‚Äî Trace `Decoded` fan-out

### Symptom

`TraceView` has a `Decoded` column bound to `TraceEntry.Decoded` (`src/PeakCan.Host.App/Views/TraceView.xaml:66`) but the column is always empty.

`TraceEntry.Decoded` is set as `init`-only (`TraceEntry.cs:39`):
```csharp
public string Decoded { get; init; } = "";
```

`TraceViewModel.AppendBatchAsync` (`TraceViewModel.cs:154-169`) never assigns it:
```csharp
Entries.Add(new TraceEntry { Timestamp, Channel, Id, Dlc, DataHex, IsError, IsFd });
// ‚Üë no Decoded assignment
```

`DbcDecodeBackgroundService` (`src/PeakCan.Host.App/Services/DbcDecodeBackgroundService.cs:67-112`) decodes frames and pushes results into `SignalViewModel.ApplyFrame` but never references `TraceViewModel` or any `TraceEntry`.

The class comment on `TraceEntry.Decoded` says: *"empty until a DBC is loaded (Task 15)"* ‚Äî the original v0.x task that promised this column was never finished. The UI was drawn but the data path was never wired.

### Architecture decision

Follow `DbcDecodeBackgroundService` to also fan out to `TraceViewModel`. **Do not** put DBC lookup into `TraceViewModel.AppendBatchAsync`:

- The offload exists for a reason: `ChannelRouter` fires frames at ~8 kbps; any per-frame work on the read loop path is budget we do not have.
- `DbcDecodeBackgroundService` is the single owner of the per-frame DBC lookup + decode logic. Adding a fan-out target is symmetric with the existing SignalViewModel fan-out.
- Avoids `TraceViewModel` taking a `DbcService` dependency, which would mix the view-model layer with a core service.

`key` = `(Id.Raw, Timestamp.Microseconds, Channel.Handle)` to disambiguate concurrent frames.

### Implementation

1. **`TraceEntry.Decoded`**: change from `init` to `set` + `PropertyChanged`:
   ```csharp
   private string _decoded = "";
   public string Decoded
   {
       get => _decoded;
       set
       {
           if (_decoded != value) { _decoded = value; PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(nameof(Decoded))); }
       }
   }
   ```
   Other `init` properties stay init-only (they are set once at construction and never change).

2. **`TraceViewModel`**: maintain a pending map:
   ```csharp
   private readonly Dictionary<TraceEntryKey, TraceEntry> _pendingDecode = new();
   public IReadOnlyDictionary<TraceEntryKey, TraceEntry> PendingDecode => _pendingDecode;
   ```
   - In `AppendBatchAsync`, after `Entries.Add(entry)`, do `_pendingDecode[key] = entry;`.
   - The dictionary is exposed via property (read-only view) for `DbcDecodeBackgroundService` to consume ‚Äî avoids direct mutation cross-thread.

3. **`DbcDecodeBackgroundService`**: add `TraceViewModel` dependency; in worker loop, after `doc.MessagesById.TryGetValue(lookupId, out var msg)`, also check `_traceVm.PendingDecode.TryGetValue(key, out var entry)` and if found, dispatch `entry.Decoded = FormatDecoded(frame, msg)` via the WPF dispatcher.

4. **Decoded string format**: `"{SignalA=42.5, SignalB=0x0A, ...}"` ‚Äî comma-separated `Name=Value` pairs. Empty string when msg has zero signals (e.g. a CAN frame with no DBC signal mapping).

### Memory & thread-safety

- `_pendingDecode` is mutated only from the UI thread (inside `AppendBatchAsync`'s dispatcher action).
- `DbcDecodeBackgroundService` worker reads `_pendingDecode` via the read-only interface; **does not mutate it directly**.
- Decoded field is mutated via the dispatcher (`entry.Decoded = ...` inside `Dispatcher.InvokeAsync`) ‚Äî preserves the existing "view-model properties change only on UI thread" rule.

### Pending-map memory bound

Max rows in Trace is `MaxRows = 1_000` (default). Worst-case pending map = 1 000 entries. Each entry ‚âà 32 B (key struct + reference) ‚Üí ~32 KB. Negligible. The pending entry is removed when the DBC worker fills `Decoded`; for frames not in the DBC, the map entry remains until the entry is trimmed from `Entries` by the FIFO cap. The trim path removes from `_pendingDecode` alongside `Entries.RemoveAt(0)`.

### Out of scope for Item 2

- Decoded for RTR / Error frames (DbcService only matches data frames)
- Decoded for CAN FD with BRS / ESI (DBC-level decode ignores bit-rate flags)

## Item 4 ‚Äî `SendView` adds RTR / BRS / ESI checkboxes

### Symptom

`CanFrame.Flags` (`src/PeakCan.Host.Core/FrameFlags.cs`) carries:
- `Rtr` (bit 0) ‚Äî Remote Transmission Request
- `BitRateSwitch` (bit 1) ‚Äî CAN FD data-phase higher baud
- `ErrorStateIndicator` (bit 2) ‚Äî CAN FD transmitter error-passive

`SendViewModel.SendAsync` (`SendViewModel.cs:87`) only sets `Fd`:
```csharp
var flags = IsFd ? FrameFlags.Fd : FrameFlags.None;
```

The other three bits are silently dropped on outbound frames.

### Fix

1. Add three `[ObservableProperty]` booleans to `SendViewModel`: `IsRtr`, `IsBitRateSwitch`, `IsErrorStateIndicator`.
2. Extend the flags builder:
   ```csharp
   var flags = FrameFlags.None;
   if (IsFd) flags |= FrameFlags.Fd;
   if (IsRtr) flags |= FrameFlags.Rtr;
   if (IsBitRateSwitch) flags |= FrameFlags.BitRateSwitch;
   if (IsErrorStateIndicator) flags |= FrameFlags.ErrorStateIndicator;
   ```
3. `SendView.xaml`: add three `CheckBox` controls next to the existing "Extended" and "CAN FD" checkboxes:
   - "RTR" ‚Äî disabled when `IsFd == true` (RTR is CAN 2.0 only, not valid with FD)
   - "BRS" ‚Äî disabled when `IsFd == false`
   - "ESI" ‚Äî disabled when `IsFd == false`
4. `SendAsync` validation: if `IsRtr && IsFd`, set `Status = "RTR is not valid for CAN FD"` and abort.

## Item 5 ‚Äî Send frame library

### Symptom

OEM workstations typically send a small set of well-known frames repeatedly (door unlock cmd, ignition on, diagnostic request). Users currently re-type ID + Data + flags every time.

### Design

New class `SendFrameLibrary` (`src/PeakCan.Host.App/Services/SendFrameLibrary.cs`):
- Storage: `%APPDATA%\PeakCan.Host\send-library.json` (single file, schema v1).
- Atomic write: write to `*.tmp` then `File.Move(..., overwrite: true)` ‚Äî prevents partial files if app crashes mid-save.
- API:
  ```csharp
  public sealed class SendFrameLibrary
  {
      public IReadOnlyList<SavedFrame> Load();
      public void Save(IEnumerable<SavedFrame> frames);
      public sealed record SavedFrame(
          string Name,
          uint RawId,
          bool IsExtended,
          bool IsFd,
          bool IsRtr,
          bool BitRateSwitch,
          string DataHex,
          DateTimeOffset SavedAt);
  }
  ```
- `Load()` semantics:
  - File missing ‚Üí return empty list, log `Information`.
  - JSON parse failure ‚Üí log `Error` with path + exception, return empty list. **Do not throw** ‚Äî corrupted library must not block SendView.
- `Save()` semantics:
  - Atomic write (`.tmp` then `File.Move(..., overwrite: true)`). Overwrite existing file.
  - I/O failure (disk full, permission denied) ‚Üí throws `IOException` / `UnauthorizedAccessException`. The caller ‚Äî `SaveCurrentToLibraryCommand` ‚Äî catches and sets `Status` accordingly. (JSON serialization of an in-memory `SavedFrame` cannot fail, so no JSON-level error path exists.)

### JSON schema v1

```json
{
  "version": 1,
  "frames": [
    {
      "name": "Door Unlock",
      "rawId": "0x100",
      "isExtended": false,
      "isFd": false,
      "isRtr": false,
      "bitRateSwitch": false,
      "dataHex": "DEADBEEF",
      "savedAt": "2026-06-26T14:30:00+08:00"
    }
  ]
}
```

### UI integration

`SendViewModel` additions:
```csharp
[ObservableProperty] private ObservableCollection<SavedFrame> _library = new();
[ObservableProperty] private SavedFrame? _selectedLibraryFrame;
[RelayCommand] private void RefreshLibrary();   // calls _libraryService.Load() ‚Üí assigns _library
[RelayCommand] private void SaveCurrentToLibrary();   // prompts name, appends, RefreshLibrary
[RelayCommand] private void LoadFromLibrary(SavedFrame f);   // populates IdText, DataText, IsExtended, IsFd, IsRtr, IsBitRateSwitch
[RelayCommand] private void DeleteFromLibrary(SavedFrame f); // removes, RefreshLibrary
```

`SendView.xaml`: add a second `Expander` labelled "Frame library" with:
- `DataGrid` bound to `Library`, AutoGenerateColumns=False, columns: Name / ID / Data / Type
- "Save current" button ‚Üí opens dialog (or inline TextBox) for naming
- "Load" / "Delete" buttons (or row context menu)

## Item 6 ‚Äî Recording UI exposure

### Symptom

`RecordService` (`src/PeakCan.Host.App/Services/RecordService.cs`, 244 LoC) is registered in DI (`AppHostBuilder.cs:99`) and attached to `ChannelRouter` via `SinkWiringService` (`Composition/SinkWiringService.cs:36,48`). It supports:
- `StartRecording(path, RecordFormat format)` / `StopRecording()`
- `IsRecording` / `FrameCount`
- ASC (Vector CANalyzer-compatible) + CSV formats
- Header / Footer / per-frame write with proper Vector ASC conventions (`date ... base hex  timestamps absolute`, per-line `timestamp channel ID dlc data flags`)

But `RecordViewModel` / `RecordView` / `Record` tab do not exist. **No UI can trigger recording today.**

### Design

1. New `RecordViewModel`:
   ```csharp
   public sealed partial class RecordViewModel : ObservableObject
   {
       private readonly RecordService _record;
       [ObservableProperty] private string _outputPath;
       [ObservableProperty] private RecordFormat _format = RecordFormat.Asc;
       [ObservableProperty] private bool _isRecording;
       [ObservableProperty] private long _frameCount;
       [RelayCommand] private void Browse();   // SaveFileDialog ‚Üí OutputPath
       [RelayCommand] private void Start();    // _record.StartRecording(OutputPath, Format)
       [RelayCommand] private void Stop();     // _record.StopRecording()
   }
   ```

2. Default `OutputPath`: `%APPDATA%\PeakCan.Host\recordings\{yyyyMMdd-HHmmss}.asc`. Directory is created on demand (`Directory.CreateDirectory(Path.GetDirectoryName(OutputPath)!)` before `Start`).

3. Status display: TextBlock next to the Start/Stop buttons showing last action result:
   - Success: `"Recording ‚Üí {OutputPath} ({frameCount} frames)"`
   - Failure: `"FAIL: {ex.Message}"`

4. `IsRecording` / `FrameCount` polled every 200 ms via a `DispatcherTimer` (same pattern as Item 3 cyclic count).

5. New `RecordView.xaml` (`src/PeakCan.Host.App/Views/RecordView.xaml`):

   `AppShellView.xaml` (already uses DataTemplates to route `CurrentPanel` ‚Üí views) ‚Äî add a DataTemplate for `RecordViewModel` ‚Üí `RecordView`.

### Format default

`Format = RecordFormat.Asc` because:
- Vector ASC is the industry standard (CANalyzer / CANoe / many third-party tools read it directly)
- Round-trip with other OEM tools is the killer feature of recording in this app
- CSV is offered as a fallback for users who need spreadsheet ingestion

### Out of scope for Item 6

- Replay (parsing + sending back recorded frames) ‚Äî **deferred to v1.3.0 MINOR**
- File rotation / size cap ‚Äî **deferred** (single-file recording is sufficient for typical OEM use; can split if user reports long-running captures)
- DBC-aware recording (substituting signal names into a parallel file) ‚Äî **deferred** (DBC can change over time; ASC-with-DBC sidecar is a separate feature)
