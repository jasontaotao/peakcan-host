1. 产品简介

PeakCan Host 是一款运行在 Windows 上的 WPF 桌面应用，用于对接 PEAK PCAN-USB FD（或 Pro FD）CAN/CAN-FD 硬件，提供通用的 CAN 总线监控、解码、手动发送与总线负载统计能力。

核心能力

能力说明
多通道探测 & 连接枚举 PCAN-USB 1–16 通道，用户可选择任意通道连接
Trace 视图虚拟化 DataGrid 滚动展示所有接收到的帧（时间戳 / 通道 / ID / DLC / 十六进制数据 / 解析结果），支持 ID 前缀过滤
DBC 解码解析 .dbc 文件并填充消息表（发送方 / DLC / 信号列表），支持多路复用信号解码
Signals 视图对收到帧按 DBC 解码信号，实时显示原始值、物理值、Value-Table 解码名称
手动发送输入 CAN ID + 十六进制数据发送；支持 CAN-FD 与 11/29-bit 切换；支持周期性发送
帧录制将接收到的帧录制到 ASC（Vector ASCII）或 CSV 文件
Stats 视图1 Hz OxyPlot 折线图，显示每秒帧数与总线负载百分比，含总帧 / 错误帧计数
Script 视图 v1.2.0CodeMirror 6 脚本引擎 + Roslyn 沙箱，可在 UI 内编写/运行 C# 脚本自动化测试场景
UDS 诊断栈 v1.2.04 面板 UDS 诊断 UI（Session / DID / Routine / DTC），含 DataGrid 绑定的 JSON 配置 DID/Routine 库与共享彩色 OutputLog
滚动日志Serilog 按天滚动写入 %LocalAppData%\PeakCan.Host\logs\，保留 14 天

2. 安装与前置条件

2.1 运行已发布的 EXE（推荐）

无需任何额外依赖：

  • 操作系统：Windows 10 (1809 及以上) 或 Windows 11
  • PEAK PCAN 驱动（官方下载页）
  • PCAN-USB FD（或 Pro FD）硬件一台

下载 PeakCan.Host.exe（约 66 MB，包含 .NET 10 运行时）后双击即可启动。

2.2 从源码运行 / 编译

  • .NET 10 SDK（官方下载）
  • 同上：Windows 10/11 + PEAK 驱动 + PCAN-USB FD 硬件

克隆源码后：

dotnet build PeakCan.Host.slnx -c Release

dotnet run --project src/PeakCan.Host.App

2.3 生成自包含单文件 EXE

dotnet publish src/PeakCan.Host.App -c Release -r win-x64 \

    --self-contained true \

    -p:PublishSingleFile=true \

    -p:IncludeNativeLibrariesForSelfExtract=true \

    -p:EnableCompressionInSingleFile=true \

    -o artifacts/win-x64/



artifacts/win-x64/PeakCan.Host.exe

3. 快速开始（5 分钟跑通）

  1. 插上 PCAN-USB FD，确认 PEAK 驱动已正确安装（设备管理器可见"PEAK-System PCAN-USB FD"）。
  2. 启动 PeakCan.Host.exe，主窗口默认打开 Trace 标签页。
  3. 点击工具栏上的 Probe 按钮 → 状态栏应显示 "Detected N channel(s)"，通道下拉列表显示检测到的通道。
  4. 从通道下拉列表选择要连接的通道，点击 Connect → 状态变为 "Connected"。
  5. （可选）菜单 File ▸ Open DBC... 选择 .dbc 文件 → Signals 标签页将随帧到达实时显示解码结果。
  6. 切到 Stats 标签页，可观察 1 Hz 折线图。
  7. 切到 Send 标签页，输入 ID 与数据，点击 Send 发送一帧。
没插硬件时 Probe 会怎样？通道列表显示 "No PEAK hardware detected: ..."，Connect 按钮自动禁用。

5. Trace（实时帧追踪）

图 2 — Trace 视图（默认标签）。每行对应一帧接收事件。

5.1 字段含义

列含义
Time帧接收时刻（本地时间，HH:mm:ss.ffffff）。
Ch硬件通道名（如 USB1 / USB2 / PCAN_PCIE_1），由通道下拉的选择决定。
IDCAN ID。Standard 11-bit 显示为 0xNNN（≤ 0x7FF）；Extended 29-bit 显示为 0xNNNNNNNN。
DLC数据长度码（0-8 为经典 CAN；CAN-FD 可达 64）。
Data十六进制字节序列（无分隔符）。
Decoded若该帧 ID 命中已加载 DBC 中某条消息，且当前帧不是错误帧，则显示"信号名=物理值"的简短摘要；否则显示 —。

5.2 行为细节

  • 行上限：MaxRows = 10 000。超出后从顶部 FIFO 丢弃最旧的帧。
  • 虚拟化：行/列均启用虚拟化（Recycling），10 k 行滚动无明显卡顿。
  • 性能：典型 1 Mbps 总线下帧率约 4 000 fps 时仍能稳定滚动；超过 8 000 fps 经典上限时可能略有延迟。
  • 错误帧：错误帧也会出现在列表（IsError 标志内部可见），但默认 Decoded 列对错误帧显示 —。
提示：使用滚轮或 Page Down 键快速跳到最新帧。DataGrid 选中行不会自动跟随——这避免了视觉跳动。

6. DBC（DBC 文件载入）

图 3 — DBC 视图（菜单 File ▸ Open DBC... 或本页 Open DBC... 按钮触发文件对话框）。

6.1 操作流程

  1. 点击 Open DBC... 按钮 → 弹出文件选择对话框（过滤器：*.dbc）。
  2. 选择文件后，解析在后台线程执行，状态文本显示 "Parsing..."。
  3. 解析成功后：
    • 消息列表被清空并填充新内容（每个 BO_ 一行）。
    • 状态文本变为 "Loaded N messages from <filename>"。
    • Signals 视图的旧条目被自动清除（避免与新 DBC 不一致）。
  4. 解析失败时，状态文本显示 "FAIL: <Code> <Message>"。

6.2 字段含义

列含义
ID消息的 CAN ID（来自 BO_ 行）。
Name消息名称。
DLCBO_ 行声明的数据长度。
Sender发送节点（来自 BO_ 第二个 token）。
Signals该消息下信号总数（不含多路复用/被多路复用信号之外的复用关系——即 DBC 中 SG_ 行计数）。
注意：重复加载同一 DBC 文件是允许的；每次都会触发完整重解析并清空 Signals 视图。 不支持热重载——如需重新加载需重新打开文件。

7. Signals（信号解码视图）

图 4 — Signals 视图：每帧解码后按 (Message, Signal) upsert，保证滚动状态稳定。

7.1 字段含义

列含义
Message所属消息名（来自 DBC BO_）。
Signal信号名（来自 SG_）。
Raw未应用 factor/offset 的整数值（0xN 格式）。
Physical应用 factor/offset 后的物理值（最多 3 位小数）。
Unit信号单位（来自 DBC SG_ 的单位 token；若为空则显示 —）。

7.2 行为细节

  • 前提：必须先在 DBC 视图加载一个 DBC 文件，否则 Signals 视图始终为空。
  • 解码规则：physical = raw * factor + offset。Intel/ Motorola 字节序、signed/unsigned、有符号数都按 DBC 规则解码。
  • 多路复用：M / m 多路复用信号在 v1.1.0 起完整解码；当前多路复用器值下激活的子信号会显示在 Signals 视图。
  • 更新频率：随帧到达实时更新；满负载下界面帧率受 WPF DataGrid 限制。
  • 跨线程：SDK 读线程触发解码，结果通过 WPF Dispatcher 投递到 UI 线程（你不会看到 NotSupportedException）。
为什么每行"原地更新"而不是追加？追加会让 DataGrid 反复创建/销毁行容器，引发滚动跳变与回收抖动。按 (Message, Signal) 原地替换既保留了最新值，又避免了视觉抖动。

8. Send（手动发送帧）

图 5 — Send 视图。ID 与 Data 均接受空格 / 连字符 / 大小写不敏感。

8.1 输入规则

字段规则
ID (hex)十六进制；非 Extended 模式最大 0x7FF（11 位），Extended 模式最大 0x1FFFFFFF（29 位）。越界或非十六进制时 Send 仍可点，但状态行显示 "Invalid ID: ..."，不会真的发送到总线。
Extended勾选 = 29-bit 帧格式；不勾 = 11-bit。
CAN FD勾选 = 设置 FDF 标志位（DLC 可达 64）；不勾 = 经典 CAN（DLC 0-8）。
Data (hex)十六进制字节序列；允许插入空格与连字符作分隔符（"DE AD-BE EF" 等价于 "DEADBEEF"）；奇数长度自动前补 0（"ABC" = {0x0A, 0xBC}）；空字符串或纯分隔符会被拒绝（显示 "Invalid data: ..."）。

8.2 操作流程

  1. 确保状态栏显示 "Connected"——未连接时 Send 仍可点但发送会失败。
  2. 填写 ID 与 Data，必要时切换 Extended / CAN FD。
  3. 点击 Send。
  4. 状态行更新：
    • 成功 → "Sent N bytes @ 0x<CanId>"（深绿）。
    • 失败 → "FAIL: <Code> <Message>"（深红）。
安全提示：手动发送会真实地发到 CAN 总线上。请确认已连接目标 ECU 的总线或具备终止电阻的总线段，避免对真实车辆/设备造成意外干扰。

8.3 错误码示例

场景状态行输出
未连接就点 SendFAIL: NotConnected ...
硬件错误（线缆断开等）FAIL: BusOff ... 或 FAIL: DriverError ...
DLC 越界（经典 CAN DLC > 8）FAIL: ParameterInvalid ...

9. Stats（总线统计）

图 6 — Stats 视图。蓝色 = FPS，红色 = 总线负载百分比；上为标题，下为 Sample 索引。

9.1 字段含义

元素含义
Total frames自连接以来累计收到的所有帧数（含错误帧）。
Error frames上述累计中被标记为 IsError = true 的子集。
Frames / sec（蓝线）过去 1 秒内收到的帧数。1 Hz 采样。
Bus load %（红线）过去 1 秒内总线占用率（由 PEAK SDK 报告）。

9.2 行为细节

  • 刷新频率：1 Hz（StatisticsService 后台定时器）。
  • 窗口大小：60 个样本 = 1 分钟历史；超出后丢弃最旧样本。
  • Y 轴：0..100（统一量程方便比对；FPS 在满负载时会"贴顶"，但用于看趋势而非绝对值）。
  • 跨线程：定时器线程产生快照，通过 Dispatcher 跳到 UI 线程更新集合，WPF 自动刷新图表。
提示：典型 1 Mbps 总线理论上限约 8000 fps；超过 50% 负载意味着总线接近饱和，需要注意响应延迟。

10. Script（脚本自动化）

Script 视图（v1.2.0 起）内嵌 CodeMirror 6 编辑器 + Roslyn 沙箱，允许在 UI 内编写/运行 C# 脚本，自动化重复的 CAN 总线测试场景（例：周期发帧、断言特定帧到达、清空 DTC）。

10.1 典型用法

  • 周期性发送：Send.PeriodicAsync(canId, data, intervalMs) 启动后台定时器。
  • 帧断言：订阅 Trace.Frames 集合变更，在指定时间窗内断言期望帧到达。
  • UDS 流程编排：调用 UdsClient.DiagnosticSessionControlAsync / ReadDataByIdentifierAsync 等，与 UDS 视图共享同一个 UdsClient 实例。

10.2 沙箱约束

安全模型：脚本在独立 AssemblyLoadContext + 受限 Permissionset 中执行，无法访问文件系统、网络、注册表。允许的 API：Send / Uds / Trace / Stats / Script 命名空间下的命令对象。

脚本的写入/读取位于 %LocalAppData%\PeakCan.Host\scripts\，文件名 *.csx。运行结果实时回写到右下角的输出面板（与 UDS 视图的 RichTextBox OutputLog 风格一致）。

11. UDS（UDS 诊断栈）

UDS 视图（v1.2.0 起）是 PeakCan Host 的核心诊断 UI，按 ISO 14229（UDS-on-IP）实现应用层协议，运行在 CAN 之上。所有命令通过当前连接（UdsClient 单例）发往 ECU。

图 7 — UDS 视图：顶部 Session 条 + 3-tab TabControl + 底部共享 OutputLog。

11.1 Session 条（顶部）

控件行为
Default 按钮DiagnosticSessionControl 0x01 — 切到默认会话。
Extended 按钮DiagnosticSessionControl 0x02 — 切到扩展诊断会话（解锁 DID Read/Write）。
Programming 按钮DiagnosticSessionControl 0x03 — 切到编程会话（预刷写状态）。
TesterPresent 复选框勾选 = 后台每 2 秒发送 TesterPresent 0x3E 0x00 维持非默认会话；取消勾选停止循环。Dispose 被显式调用时也会停掉循环并复位标志位——但 DI 容器不调用 Dispose（单例 VM 生命周期 ≈ 进程），详见 15.2。
SecurityAccess (Level 1) 按钮触发 SecurityAccess RequestSeed 0x01 → 调 IKeyDerivationAlgorithm.ComputeKey(...) → SendKey 0x02。成功时 Level: 0x01 显示在右侧。详见 §11.6。
Session: 文本当前会话名（Default / Extended / Programming）。
Level: 0xNN 文本当前认证等级。未认证时显示 (not authenticated)。

11.2 DIDs Tab

DataGrid 显示 DID 库（默认内置 5 个：VIN / PartNumber / SupplierID / HardwareVersion / SoftwareVersion），可由用户 JSON 文件扩展。

11.2.1 DID 数据源

DID 库按以下优先级合并：

  1. 内置默认（DidDatabase.BuiltInDefaults()）：5 个标准 VIN/PartNumber 等。
  2. 用户 JSON：%LOCALAPPDATA%\PeakCan.Host\uds-dids.json（可选）。同名 ID 覆盖内置，缺失追加。

JSON 格式示例：

{

  "dids": [

    { "id": 49152, "name": "BatteryVoltage", "lengthBytes": 2, "writable": false },

    { "id": 49153, "name": "CoolantTemp",   "lengthBytes": 1, "writable": false }

  ]

}

JSON 文件缺失或格式错误时不会抛异常——回退到内置默认，并在 Serilog 日志写一条 Warn。

11.2.2 DIDs Tab 字段

列含义
IDDID 标识符（16-bit，hex 显示 0xNNNN）。
Name人类可读名。
Length数据长度（字节）。
R/W是否可写（来自 JSON 的 writable 字段，内置默认都为 R）。
Value最近一次 Read 后的字节序列（首次未读取时为空）。

11.2.3 操作流程

  1. 选择 DataGrid 中的某一行 → 右侧详情面板填充 Name / Length。
  2. 点击 Read → UdsClient.ReadDataByIdentifierAsync(id) → 成功后 Value 列填充十六进制字节。
  3. 若 DID 可写（Writable=true），可在 Write value (hex) 文本框输入字节（空格/连字符分隔），点击 Write → WriteDataByIdentifierAsync(id, bytes)。
  4. 每次操作的结果会追加到底部 OutputLog：Info（成功）/ Warn（NRC）/ Error（异常）。

11.3 Routines Tab

DataGrid 显示 Routine 库（默认内置空数组），由用户 JSON 文件 %LOCALAPPDATA%\PeakCan.Host\uds-routines.json 加载（结构与 DIDs 类似，字段 id / name）。

列含义
IDRoutine 标识符（16-bit hex）。
NameRoutine 名（来自 JSON）。
Status当前状态（Idle / Running / Stopped）。
Result最近一次 Query 的结果字节序列。

操作：Start（RoutineControl 0x01）/ Stop（0x02）/ Query（0x03）。结果写入 OutputLog + DataGrid 的 Status / Result 列。

11.4 DTCs Tab

DTC 不走 JSON 文件——直接通过 ReadDtcInformationAsync(subFunction=0x02, statusMask=0xFF) 从 ECU 拉取。

列含义
Code3 字节 DTC 代码（0xNNNNNN）。
Status1 字节 DTC 状态位掩码（0xNN）。
Description分类描述（Powertrain / Chassis / Body / Network / Unknown，按 ISO 15031-6 高 2 bit 分类）。

操作：Read DTCs 拉取并填充 DataGrid；Clear DTCs 发送 ClearDiagnosticInformation(0xFFFFFF) 并清空本地列表。

11.5 ODX 诊断描述导入（v2.0.0 起，v2.0.4 支持 .odx-d）

v2.0.0 起，UDS 视图顶部条新增 Load ODX… 按钮。v2.0.4 起支持 Vector CANdelaStudio 输出的 .odx-d 文件（ODX-D 子集）：

  1. 点击按钮 → 文件对话框（filter: *.odx / *.pdx / *.odx-d / *.*）。
  2. 选择供应商提供的 .odx、.pdx（zip 容器）或 .odx-d（Vector CANdelaStudio 导出）文件。
  3. OdxImportService 解析 DIAG-LAYER-CONTAINER → 映射 DOP-BASE / DTC-DOP / ECU-JOB（canonical ODX 2.x）+ BASE-VARIANT + REQUEST（0x22/0x2E/0x31 三个 UDS service id，.odx-d 实际数据所在）→ 填充 DidDatabase / DtcDatabase / RoutineDatabase。
  4. 底部 status bar 显示 ODX loaded: N DIDs / M Routines / K DTCs from <path>。如果 warnings > 0，会附 warning 数（也显示在 modal summary）。
  5. DTC panel 重新刷新 — 如果之前 ECU 已读过 DTC 但描述为空，现在会自动用 ODX 中的 TEXT/DTC-TAB 替换 ISO 14294 category fallback。

v2.0.5 起：DID 数据长度（LengthBytes）通过走 POS-RESPONSE → DOP-REF → DIAG-CODED-TYPE BIT-LENGTH 链自动解析（响应体总 bit-length 求和后按 byte 对齐）。DID 列表、Writable 标志、LengthBytes 全部正确导入。

11.6 SecurityAccess 与 IKeyDerivationAlgorithm

SecurityAccess（Level 1）调用流程：

  1. UI 点击按钮 → UdsClient.SecurityAccessAsync(0x01)。
  2. Core 层先发 RequestSeed 0x01，收到 ECU 发的 seed。
  3. 调用已注册的 IKeyDerivationAlgorithm.ComputeKey(seed, level) 计算 key。
  4. 发 SendKey 0x02，等 ECU 响应正/负响应。
关键：当前默认注册的是 PlaceholderKeyAlgorithm——它原样回传 seed 作为 key，仅用于联调与冒烟测试。任何真实 ECU 都会拒绝（NRC 0x35 invalidKey）。

OEM 客户需要实现自己的 IKeyDerivationAlgorithm（接口在 PeakCan.Host.Core/Uds/IKeyDerivationAlgorithm.cs），并在 AppHostBuilder 中以单例注册替换占位实现。详见 附录 A §A.2。

如果未注册任何算法（或注册的是占位实现），UI 会输出：

14:23:05.001  Warn   Security access failed: KeyAlgorithmNotConfiguredException

14:23:05.002  Info   Hint: register an IKeyDerivationAlgorithm implementation in DI before invoking SecurityAccess.

12. Trace Viewer（离线 .asc 回放） v3.0.0

Trace Viewer（v3.0.0 MINOR 起）是 PeakCan Host 的离线回放窗口， 对应中文"回放"语义：加载已录制的 .asc 文件并按时间轴只读 地查看信号变化曲线，绝不向 CAN 总线写入任何帧。

关键区分：
  • Trace Viewer（回放，v3.0.0）：加载 .asc → 解析 → 显示图表。 无总线写入。代码路径走 ITraceViewerService，反射测试 Constructor_DoesNotAcceptIReplayFrameSink 强制该类型不能引用 IReplayFrameSink。
  • Replay（重发，v1.4.0）：加载 .asc → 通过 IReplayFrameSink.SendFrameAsync 把帧重新写到总线上，影响真实总线状态。自 v2.1.4 PATCH 起 View ▸ Replay 菜单入口可用。

12.1 启动方式

  1. 主窗口菜单 View ▸ Trace Viewer…（AppShellViewModel.ShowTraceViewerCommand）。
  2. 弹出非模态窗口（TraceViewerView），可与主窗口同时操作、独立关闭。
  3. 窗口内顶部工具栏：Open .asc（打开录制文件）+ Open DBC（可选，解析信号）+ 播放控件 ▶ / ⏸ / ⏹ + 速度下拉（0.25× / 0.5× / 1× / 2× / 4×）+ Loop 复选框 + 进度 scrubber。
  4. 窗口中部：横向 split（GridSplitter）。
    • 左侧：信号列表（按 (CAN ID, signal) 一行），列：Plot 复选框 / CAN ID hex / 信号名 / 最新值 / 单位。
    • 右侧：每个选中信号的子图（每个 OxyPlot.PlotView 独立 Y 轴），共享 X 轴（0..TotalDuration）。
  5. 窗口底部：状态栏 + Export CSV（导出所有已勾选信号样本，X = 时间，列为各信号值）。

12.2 关键行为

行为说明
无总线写入 TraceViewerService 不持有 IReplayFrameSink； 反射测试 Constructor_DoesNotAcceptIReplayFrameSink 在每次 CI 中验证 IReplayFrameSink 不出现在 TraceViewerService 的 ctor 参数、属性或字段中。 加载 ASC 后，CPU 资源被消耗，但 PCAN-USB FD 硬件不会发出任何帧。
帧数上限 软警告 50 000 帧；硬上限 100 000 帧。超出后 TraceViewerService.LoadAsync 抛 ReplayLoadException，UI 弹错误对话框。
共享 X 轴同步 所有子图的 X 轴通过 LinearAxis.AxisChanged 事件级联； 用 _syncing 布尔标志位防止重入 ping-pong。zoom / pan 一个子图时其他子图同步。
播放游标 红色垂直 LineAnnotation，由 ValueChanged 事件驱动；为避免 10k 点的全量重绘， 用 PlotModel.InvalidatePlot(false) 只刷新 annotation 层（~50× 快于无参重载）。
CSV 导出 所有 Plot 复选框勾选的信号 → 写出 time,SignalA,SignalB,… 格式 CSV； 时间列单位为秒（相对录制起点）。
统计 chip 每个子图底部显示 min / max / avg / n 四个 chip，随 playback 实时刷新。
DBC 不必加载 无 DBC 时按 ID 列出原始帧（hex）；有 DBC 时按 (ID, signal) 展开成可绘制的物理值。
多实例 可同时开多个 Trace Viewer 窗口看不同 .asc；每个窗口独立 VM（DI 单例复用，但 View 独立）。

维度Trace Viewer（v3.0.0）Replay（v1.4.0 / v2.1.4）

安全提示：Trace Viewer 是只读工具，不会对总线产生任何影响；但若你同时打开了 Replay（重发）窗口，请确认 Replay 的目标通道不是你正在观察的同一条总线——否则重发帧会与录制回放的图表 "混入"，造成分析误导。两个窗口的视觉样式（颜色、布局）有显著差异，但功能职责不同。

12.5 已知限制（V1）

  • 最多同时显示 10 个信号子图（Tableau 10 调色板）；超过 10 个需手动取消勾选前面的再勾选新的。
  • 录制时间超过 100 000 帧会被拒绝加载；建议先用 Record 服务的 CSV 模式导出大文件做趋势分析。
  • DBC 热重载不支持——换 DBC 后需关闭 + 重开窗口。
  • 无书签 / 标注 / 多 trace 对比 / PNG 导出 / J1939 协议解码（均明确出 V1 scope）。

15. FAQ & 已知限制

15.1 FAQ

Q1: 能否连接多块 PCAN-USB FD 或多通道?

A: 可以。v1.1.0 起 Probe 枚举 PCAN-USB 1–16 通道，工具栏的通道下拉显示所有检测到的通道，可任意选择后 Connect。当前限制：同一时刻单连接（多通道并行收发不在 v3.x 范围——见 §15.3 路线图）。

Q2: 能否录制总线数据供后续回放?

A: 录制 可以（v1.1.0 起，Record 服务支持 ASC Vector ASCII + CSV 输出）。回放已提供：v1.4.0 MINOR 起 Replay 服务实现（Open/Play/Pause/Stop/Scrubber/Speed/Loop/CAN-ID 过滤/时间范围过滤），但 UI 入口在 v2.1.4 PATCH 之前未挂上 AppShell。自 v2.1.4 PATCH 起 View ▸ Replay 菜单可直接打开。

Q3: 多路复用（M / m）信号能否解码?

A: 可以。v1.1.0 起 DbcParser 识别 M / m 修饰符，SignalDecoder 按多路复用器值动态选择被复用信号。Signals 视图会显示当前多路复用器值下激活的子信号。

Q4: 能否在后台运行脚本自动化测试?

A: 可以。v1.2.0 起内置 Script 视图：CodeMirror 6 编辑器 + Roslyn 沙箱，%LocalAppData%\PeakCan.Host\scripts\*.csx 文件管理。详见 §10。

Q5: 程序崩溃了日志在哪?

A: %LocalAppData%\PeakCan.Host\logs\peak-yyyy-MM-dd.log（Serilog，保留 14 天），加上 Windows 应用事件查看器（Application Error 事件）。

Q6: 能否选择非 1 Mbps 的波特率?

A: 可以。v1.1.0 起工具栏的波特率下拉支持 1 Mbps / 500 kbps / 250 kbps / 125 kbps / 100 kbps / 50 kbps / 等常见档位（AvailableBaudRates 列表由 PEAK SDK 提供）。

Q7: UDS SecurityAccess 直接能工作吗?

A: 不能直接。v1.2.0 默认注册的是 PlaceholderKeyAlgorithm——原样回传 seed，仅用于联调；真实 ECU 会返回 NRC 0x35 invalidKey。OEM 客户需要实现 IKeyDerivationAlgorithm 并在 AppHostBuilder 替换。详见 §11.6。

Q8: DID/Routine 库能不能扩展?

A: 可以。v1.1.0 起 %LOCALAPPDATA%\PeakCan.Host\uds-dids.json + uds-routines.json 可放置用户 JSON，缺失/格式错误时回退到内置默认。详见 §11。

Q9: UDS 视图如何与 Trace/DBC 视图联动?

A: 通过同一个 UdsClient 单例（DI 注入）。UDS 视图发出的请求帧会通过底层 CAN 通道被 ChannelRouter 广播，Trace 视图同步看到；DBC 中已加载的消息 ID 在 UDS 响应解码时被复用（如果响应 ID 命中 DBC 中消息）。

Q10: 能否在 UI 关闭后保留 TesterPresent?

A: 不能。TesterPresent 后台循环由 SessionPanelViewModel 持有，VM 是 DI 单例直到进程退出。关闭 UDS 视图标签不会停止 TesterPresent；切换会话前请手动取消勾选，或关闭整个应用。

A.2 关键决策

决策理由
.NET 10（非 8）开发机仅有 10.0.300 SDK；自包含发布后目标机不需任何运行时。
Peak.PCANBasic.NET 5.0.1（非 Peak.Can.Basic）旧的 Peak.Can.Basic 在 nuget.org 上不可寻；Peak.PCANBasic.NET 是 PEAK 官方替代（12.7 万次下载）。
OxyPlot.Wpf 2.2.0（非 LiveChartsCore 2.0.4）LiveCharts 2.0.4 的原生依赖（OpenTK + SkiaSharp.Views.WPF）仅目标 .NET Framework，在 .NET 10 上启动失败。OxyPlot 是纯托管的，干净。
多通道探测 + 单连接（v1.1 起）v0.x MVP 仅探测 + 连接第一个 handle 0x51；v1.1 起 Probe 枚举 1–16 通道并填充下拉，但同一时刻仍只允许单连接。多通道并行收发不实现（v3.x 范围外的长期非目标，与 J1939/CANopen/SocketCAN 一起推迟到更远的将来）。
VM 部分 IDisposable（v1.2.2 起）多数 VM 仍是 DI 单例不实现 IDisposable；但 SessionPanelViewModel 持有 TesterPresent 后台 CTS，v1.2.2 起实现 IDisposable 复位 TesterPresentActive=false + Cancel/Dispose CTS。注：当前 DI 容器不调用 Dispose()——VM 生命周期 ≈ 进程生命周期，Dispose 仅作为防御性 API 暴露。
IFileDialogService 抽象（v1.1 起）v0.x DbcViewModel.OpenAsync 内联调用 OpenFileDialog.ShowDialog，用户取消路径测试为 [Fact(Skip=...)]。v1.1 起抽取为 IFileDialogService 接口 + WpfFileDialogService 实现，所有用户取消测试改为真实断言。
IKeyDerivationAlgorithm 抽象 + PlaceholderKeyAlgorithm（v1.1 起）UDS SecurityAccess 的 seed-to-key 算法因 OEM 而异（典型：HMAC-SHA256、HSM 模拟、硬件安全模块等）。v1.1 抽取为接口并提供占位实现（原样回传 seed）让 UI 联调可跑通；OEM 客户需自行实现并替换。详见 §11.6。
UDS 4 面板 orchestrator 模式（v1.2.0 起）原 UdsViewModel 单体 279 行 / 8 个 [RelayCommand]，v1.2.0 拆为 4 个 panel VM（Session / DID / Routine / DTC） + ≤80 行 orchestrator。每个 panel VM 通过 IUdsPanel.AttachLog(...) 接口共享一个 OutputLog。
