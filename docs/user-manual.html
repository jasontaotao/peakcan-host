1. 产品简介

PeakCan Host 是一款运行在 Windows 上的 WPF 桌面应用，用于对接 PEAK PCAN-USB FD（或 Pro FD）CAN/CAN-FD 硬件，提供通用的 CAN 总线监控、解码、手动发送与总线负载统计能力。

核心能力

能力说明
硬件探测 & 连接检测 PCAN-USB FD 是否在线，以 CAN-FD 1 Mbps 打开第一路通道并接入内部帧路由器
Trace 视图虚拟化 DataGrid 滚动展示所有接收到的帧（时间戳 / 通道 / ID / DLC / 十六进制数据 / 解析结果）
DBC 解码解析 .dbc 文件并填充消息表（发送方 / DLC / 信号列表）
Signals 视图对收到帧按 DBC 解码信号，实时显示原始值与物理值（已应用 factor / offset）
手动发送输入 CAN ID + 十六进制数据发送；支持 CAN-FD 与 11/29-bit 切换
Stats 视图1 Hz OxyPlot 折线图，显示每秒帧数与总线负载百分比，含总帧 / 错误帧计数
滚动日志Serilog 按天滚动写入 %LocalAppData%\PeakCan.Host\logs\，保留 14 天
当前版本 v0.1.0 为 MVP，仅支持单通道（PCAN-USB FD 第一个 handle = 0x51）。 多通道枚举、ASC/CSV 录制、帧过滤、循环发送、多路复用信号解码均在 v1.1 路线图中， 详见 第 12 节。

2. 安装与前置条件

2.1 运行已发布的 EXE（推荐）

无需任何额外依赖：

  • 操作系统：Windows 10 (1809 及以上) 或 Windows 11
  • PEAK PCAN 驱动（官方下载页）
  • PCAN-USB FD（或 Pro FD）硬件一台

下载 PeakCan.Host.exe（约 66 MB，包含 .NET 10 运行时）后双击即可启动。

2.2 从源码运行 / 编译

  • .NET 10 SDK（官方下载）
  • 同上：Windows 10/11 + PEAK 驱动 + PCAN-USB FD 硬件

克隆源码后：

dotnet build PeakCan.Host.slnx -c Release
dotnet run --project src/PeakCan.Host.App

2.3 生成自包含单文件 EXE

dotnet publish src/PeakCan.Host.App -c Release -r win-x64 \
    --self-contained true \
    -p:PublishSingleFile=true \
    -p:IncludeNativeLibrariesForSelfExtract=true \
    -p:EnableCompressionInSingleFile=true \
    -o artifacts/win-x64/

artifacts/win-x64/PeakCan.Host.exe

3. 快速开始（5 分钟跑通）

  1. 插上 PCAN-USB FD，确认 PEAK 驱动已正确安装（设备管理器可见"PEAK-System PCAN-USB FD"）。
  2. 启动 PeakCan.Host.exe，主窗口默认打开 Trace 标签页。
  3. 点击工具栏上的 Probe 按钮 → 状态栏应显示 "USB1 (1 Mbps default)"，通道下拉显示同样的提示（如图 1 中状态栏所示）。
  4. 点击 Connect → 状态变为 "Connected to USB1 (CAN FD 1 Mbps)"。
  5. （可选）菜单 File ▸ Open DBC... 选择 .dbc 文件 → Signals 标签页将随帧到达实时显示解码结果。
  6. 切到 Stats 标签页，可观察 1 Hz 折线图。
  7. 切到 Send 标签页，输入 ID 与数据，点击 Send 发送一帧。
没插硬件时 Probe 会怎样？通道列表显示 "No PEAK hardware detected: ..."，Connect 按钮自动禁用。

4. 主窗口总览

图 1 — 主窗口布局（菜单、工具栏、状态栏）。View 菜单切换主区域。

4.1 工具栏说明

控件行为
通道列表（左侧文本）显示最近一次 Probe 的结果。初始为 "(click Probe to detect)"；成功时变为 "USB1 (1 Mbps default)"；失败时变为 "No PEAK hardware detected: ..."。
Probe 按钮检测 PCAN-USB FD 是否在线。仅在未连接时可用（!IsConnected）。
Connect 按钮以 CAN-FD 1 Mbps 打开通道 0x51。仅在 Probe 成功且未连接时可用。
Disconnect 按钮 （v0.2.0 新增）释放硬件（PCANBasic.Uninitialize），从 ChannelRouter 注销通道，清空 SendService.ActiveChannel。仅在已连接时可用（IsConnected）。可重复点击，第二次调用为 no-op（幂等）。
连接状态文本实时显示 "Disconnected" / "Connecting..." / "Connected to USB1 (CAN FD 1 Mbps)" / "Disconnecting..."（断开过程中）。

4.2 状态栏说明

底部状态栏显示当前操作的简短反馈（如 "Connected"、"FAIL: ..."）。 完整的诊断信息写入 Serilog 滚动文件，详见 第 11 节。

4.3 View 菜单

5 个标签对应 5 个视图（详见后续章节）。视图实例被缓存——重复切换不会丢失滚动位置或选中行。

5. Trace（实时帧追踪）

图 2 — Trace 视图（默认标签）。每行对应一帧接收事件。

5.1 字段含义

列含义
Time帧接收时刻（本地时间，HH:mm:ss.ffffff）。
Ch硬件通道名，当前 v0.1.0 始终为 USB1。
IDCAN ID。Standard 11-bit 显示为 0xNNN（≤ 0x7FF）；Extended 29-bit 显示为 0xNNNNNNNN。
DLC数据长度码（0-8 为经典 CAN；CAN-FD 可达 64）。
Data十六进制字节序列（无分隔符）。
Decoded若该帧 ID 命中已加载 DBC 中某条消息，且当前帧不是错误帧，则显示"信号名=物理值"的简短摘要；否则显示 —。

5.2 行为细节

  • 行上限：MaxRows = 10 000。超出后从顶部 FIFO 丢弃最旧的帧。
  • 虚拟化：行/列均启用虚拟化（Recycling），10 k 行滚动无明显卡顿。
  • 性能：典型 1 Mbps 总线下帧率约 4 000 fps 时仍能稳定滚动；超过 8 000 fps 经典上限时可能略有延迟。
  • 错误帧：错误帧也会出现在列表（IsError 标志内部可见），但默认 Decoded 列对错误帧显示 —。
提示：使用滚轮或 Page Down 键快速跳到最新帧。DataGrid 选中行不会自动跟随——这避免了视觉跳动。

6. DBC（DBC 文件载入）

图 3 — DBC 视图（菜单 File ▸ Open DBC... 或本页 Open DBC... 按钮触发文件对话框）。

6.1 操作流程

  1. 点击 Open DBC... 按钮 → 弹出文件选择对话框（过滤器：*.dbc）。
  2. 选择文件后，解析在后台线程执行，状态文本显示 "Parsing..."。
  3. 解析成功后：
    • 消息列表被清空并填充新内容（每个 BO_ 一行）。
    • 状态文本变为 "Loaded N messages from <filename>"。
    • Signals 视图的旧条目被自动清除（避免与新 DBC 不一致）。
  4. 解析失败时，状态文本显示 "FAIL: <Code> <Message>"。

6.2 字段含义

列含义
ID消息的 CAN ID（来自 BO_ 行）。
Name消息名称。
DLCBO_ 行声明的数据长度。
Sender发送节点（来自 BO_ 第二个 token）。
Signals该消息下信号总数（不含多路复用/被多路复用信号之外的复用关系——即 DBC 中 SG_ 行计数）。
注意：重复加载同一 DBC 文件是允许的；每次都会触发完整重解析并清空 Signals 视图。 不支持热重载——如需重新加载需重新打开文件。

7. Signals（信号解码视图）

图 4 — Signals 视图：每帧解码后按 (Message, Signal) upsert，保证滚动状态稳定。

7.1 字段含义

列含义
Message所属消息名（来自 DBC BO_）。
Signal信号名（来自 SG_）。
Raw未应用 factor/offset 的整数值（0xN 格式）。
Physical应用 factor/offset 后的物理值（最多 3 位小数）。
Unit信号单位（来自 DBC SG_ 的单位 token；若为空则显示 —）。

7.2 行为细节

  • 前提：必须先在 DBC 视图加载一个 DBC 文件，否则 Signals 视图始终为空。
  • 解码规则：physical = raw * factor + offset。Intel/ Motorola 字节序、signed/unsigned、有符号数都按 DBC 规则解码。
  • 跳过：多路复用信号（M / m）当前 v0.1.0 不解码，留待 v1.1。
  • 更新频率：随帧到达实时更新；满负载下界面帧率受 WPF DataGrid 限制。
  • 跨线程：SDK 读线程触发解码，结果通过 WPF Dispatcher 投递到 UI 线程（你不会看到 NotSupportedException）。
为什么每行"原地更新"而不是追加？追加会让 DataGrid 反复创建/销毁行容器，引发滚动跳变与回收抖动。按 (Message, Signal) 原地替换既保留了最新值，又避免了视觉抖动。

8. Send（手动发送帧）

图 5 — Send 视图。ID 与 Data 均接受空格 / 连字符 / 大小写不敏感。

8.1 输入规则

字段规则
ID (hex)十六进制；非 Extended 模式最大 0x7FF（11 位），Extended 模式最大 0x1FFFFFFF（29 位）。越界或非十六进制时 Send 仍可点，但状态行显示 "Invalid ID: ..."，不会真的发送到总线。
Extended勾选 = 29-bit 帧格式；不勾 = 11-bit。
CAN FD勾选 = 设置 FDF 标志位（DLC 可达 64）；不勾 = 经典 CAN（DLC 0-8）。
Data (hex)十六进制字节序列；允许插入空格与连字符作分隔符（"DE AD-BE EF" 等价于 "DEADBEEF"）；奇数长度自动前补 0（"ABC" = {0x0A, 0xBC}）；空字符串或纯分隔符会被拒绝（显示 "Invalid data: ..."）。

8.2 操作流程

  1. 确保状态栏显示 "Connected"——未连接时 Send 仍可点但发送会失败。
  2. 填写 ID 与 Data，必要时切换 Extended / CAN FD。
  3. 点击 Send。
  4. 状态行更新：
    • 成功 → "Sent N bytes @ 0x<CanId>"（深绿）。
    • 失败 → "FAIL: <Code> <Message>"（深红）。
安全提示：手动发送会真实地发到 CAN 总线上。请确认已连接目标 ECU 的总线或具备终止电阻的总线段，避免对真实车辆/设备造成意外干扰。

8.3 错误码示例

场景状态行输出
未连接就点 SendFAIL: NotConnected ...
硬件错误（线缆断开等）FAIL: BusOff ... 或 FAIL: DriverError ...
DLC 越界（经典 CAN DLC > 8）FAIL: ParameterInvalid ...

9. Stats（总线统计）

图 6 — Stats 视图。蓝色 = FPS，红色 = 总线负载百分比；上为标题，下为 Sample 索引。

9.1 字段含义

元素含义
Total frames自连接以来累计收到的所有帧数（含错误帧）。
Error frames上述累计中被标记为 IsError = true 的子集。
Frames / sec（蓝线）过去 1 秒内收到的帧数。1 Hz 采样。
Bus load %（红线）过去 1 秒内总线占用率（由 PEAK SDK 报告）。

9.2 行为细节

  • 刷新频率：1 Hz（StatisticsService 后台定时器）。
  • 窗口大小：60 个样本 = 1 分钟历史；超出后丢弃最旧样本。
  • Y 轴：0..100（统一量程方便比对；FPS 在满负载时会"贴顶"，但用于看趋势而非绝对值）。
  • 跨线程：定时器线程产生快照，通过 Dispatcher 跳到 UI 线程更新集合，WPF 自动刷新图表。
提示：典型 1 Mbps 总线理论上限约 8000 fps；超过 50% 负载意味着总线接近饱和，需要注意响应延迟。

10. DBC 解析器支持范围

v0.1.0 支持以下 DBC 关键字与特性：

类别关键字状态
元数据VERSION解析但不显示
节点定义NS_, BS_, BU_解析；节点名用于 Sender 列
消息BO_完整支持
信号SG_（含 Intel / Motorola, signed / unsigned）完整解码（除 M / m 多路复用）
值表VAL_, VAL_TABLE_解析；v1.1 在 Signals 视图显示映射名称
注释CM_解析；当前 UI 不展示
属性BA_DEF_, BA_接受；解码层忽略（无消费者）
信号组SIG_GROUP_解析；当前 UI 不展示
环境变量EV_解析；当前未使用
多路复用M / m 修饰符解析；不在 Signals 视图解码（v1.1）
Vector 扩展浮点IEEE float / IEEE double接受；解码器在关键字未识别时回退为 int
已知限制：非标准 / 厂商私有扩展可能被忽略。加载失败时状态行会显示具体错误码；常见错误为"Parse error: unexpected token"。

11. 日志与故障排查

11.1 日志位置

Serilog 按天滚动写入（保留 14 天）：

%LocalAppData%\PeakCan.Host\logs\peak-yyyy-MM-dd.log

11.2 常见问题对照表

症状可能原因建议排查步骤
Probe 显示 "No PEAK hardware detected"驱动未装 / 设备未插 / USB 接触不良检查设备管理器；重新插拔；重装 PEAK 驱动
Connect 失败 "BusOff"总线短路 / 缺少终端电阻 / 波特率不一致检查物理接线；确认对端设备使用相同波特率
Signals 视图始终为空未加载 DBC 或消息 ID 不匹配先在 DBC 视图加载 DBC；确认 Trace 中 ID 与 DBC 一致
Send 状态 "FAIL: NotConnected"未点击 Connect 或 Connect 失败先 Probe → Connect；检查状态栏
DBC 加载失败 "Parse error"DBC 含未支持的语法 / 编码用文本编辑器打开 .dbc 检查可疑行；查看日志
Trace 滚动卡顿总线负载极高（>5000 fps）预期行为；可降低总线流量或减小 MaxRows

11.3 如何获取完整诊断信息

  1. 复现问题一次。
  2. 打开日志目录：Win+R → 输入 %LocalAppData%\PeakCan.Host\logs。
  3. 找到当天日期的 peak-yyyy-MM-dd.log，把异常堆栈 + 时间戳附在 bug 报告里。

12. FAQ & 已知限制

12.1 FAQ

Q1: 能否连接两块 PCAN-USB FD？

A: 不能。v0.1.0 硬编码单通道（handle 0x51）。多通道枚举在 v1.1。

Q2: 能否录制总线数据供后续回放？

A: 不能。v0.1.0 不支持 ASC / CSV 录制。v1.1 将加入录制 + 回放。

Q3: 多路复用（M / m）信号何时能解码？

A: v1.1 路线图中。当前会被跳过，不出现在 Signals 视图。

Q4: 能否在后台运行脚本自动化测试？

A: 当前只能通过外部 .NET 控制台程序调用 PeakCan.Host.Core 解析器与 PeakCan.Host.Infrastructure SDK 适配器。v1.2 计划加入内置 CodeMirror 6 脚本引擎 + 沙箱。

Q5: 程序崩溃了日志在哪？

A: %LocalAppData%\PeakCan.Host\logs\peak-yyyy-MM-dd.log（Serilog），加上 Windows 应用事件查看器（Application Error 事件）。

Q6: 能否选择非 1 Mbps 的波特率？

A: v0.1.0 硬编码 CAN-FD 1 Mbps。其他波特率在 v1.1（计划加入下拉框）。

12.2 v0.1.0 已知限制（MVP 范围）

  • 单通道（PCAN-USB FD 第一个 handle）
  • 无 ASC / CSV 录制
  • 无帧过滤（ID / 通道过滤）
  • 无循环发送（cyclic transmission）
  • 多路复用信号（M / m）不解码
  • 无值表（VAL_）名称显示
  • 无内置脚本自动化
  • 无 UDS 诊断栈
  • 无 J1939 / CANopen 协议层
  • 仅 Windows 平台（依赖 WPF）

附录 A — 架构与关键决策

A.1 三层架构

   PeakCan.Host.App            (WPF · MVVM · BackgroundService · DI 组合根)
            │
            ▼  使用
   PeakCan.Host.Infrastructure (PEAK SDK 适配器 · ChannelRouter · BusStatistics)
            │
            ▼  使用
   PeakCan.Host.Core           (CanFrame · DBC parser · SignalDecoder · Result)

App 层被禁止直接引用 PEAK SDK（LayeringRulesTests.App_Should_Not_Depend_On_Peak_Can_Basic）； 所有硬件调用经 IChannelProbe 与 PeakCanChannel 走 Infrastructure。 CI 会拒绝破坏此边界的 PR。

A.2 关键决策

决策理由
.NET 10（非 8）开发机仅有 10.0.300 SDK；自包含发布后目标机不需任何运行时。
Peak.PCANBasic.NET 5.0.1（非 Peak.Can.Basic）旧的 Peak.Can.Basic 在 nuget.org 上不可寻；Peak.PCANBasic.NET 是 PEAK 官方替代（12.7 万次下载）。
OxyPlot.Wpf 2.2.0（非 LiveChartsCore 2.0.4）LiveCharts 2.0.4 的原生依赖（OpenTK + SkiaSharp.Views.WPF）仅目标 .NET Framework，在 .NET 10 上启动失败。OxyPlot 是纯托管的，干净。
单硬编码通道 (0x51)MVP 仅探测 + 连接第一个 handle。多通道枚举在 v1.1。
VM 非 IDisposable所有 ViewModel 是 DI 单例，整个进程生命周期存活；若实现 IDisposable 反而需要 unsubscribe 单例服务（潜在隐患）。VM 与服务一起在进程退出时消亡。
无 IFileDialogService 抽象DbcViewModel.OpenAsync 中内联调用 OpenFileDialog.ShowDialog；用户取消路径的测试为 [Fact(Skip=...)]。抽象抽取留 v1.1。
