## Root cause

`AppShellViewModel.WindowTitle` (App shell) reads `AssemblyInformationalVersion` to render the title bar prefix `PeakCan Host v{version}`. The csproj has no `<Version>` element, and the repo-root `Directory.Build.props` only sets `<Version>2.0.0</Version>` as a default. On a `feature/v3-12-0-minor` branch with N commits past the v3.8.7 tag, the .NET SDK default-fills `InformationalVersion` with `git describe` output — i.e. `v3.8.7-33-gb75317d` (an old tag + commits-since + abbreviated SHA). The user has no way to tell which PATCH version they are actually running.

This affects both:
1. **End users** — title bar shows a confusing hash instead of the version they installed.
2. **Support / bug reports** — `Assembly.GetExecutingAssembly().GetCustomAttribute<AssemblyInformationalVersionAttribute>().InformationalVersion` returns `v3.8.7-33-gb75317d`, not `v3.13.3`, so any self-reported version in logs / diagnostics is misleading.

## Fix

Create `src/Directory.Build.props` with explicit `<Version>`, `<AssemblyVersion>`, `<FileVersion>`, `<InformationalVersion>` elements all set to `3.13.3`. MSBuild auto-imports `Directory.Build.props` from the project directory and ALL ancestors; the src-level file imports AFTER the root one, so it overrides the root's `Version=2.0.0` for the App + Infrastructure projects but does NOT affect test projects under `tests/` (which never display a version anyway).

The ship script regenerates this file on each PATCH.

### Files changed

- **NEW** `src/Directory.Build.props` — hardcoded `<Version>3.13.3</Version>` (+ AssemblyVersion/FileVersion/InformationalVersion)
- **MODIFIED** `src/PeakCan.Host.App/PeakCan.Host.App.csproj` — adds a 7-line xmldoc-style comment explaining the version comes from `src/Directory.Build.props` and NOT to hardcode `<Version>` here (the ship script regenerates that file per PATCH)

### Why Directory.Build.props (not just csproj edit)

- A user who builds from source clones master + checks out `v3.13.3` tag — they get the right InformationalVersion out of the box without editing csproj.
- The ship script regenerates `src/Directory.Build.props` per PATCH, so future PATCHes (v3.13.4, v3.13.5, …) don't require manual version edits.
- Matches the v3.6.0 MINOR pattern of checking in build configuration as part of the ship.

## User-visible impact

Before: open the app → title bar reads `PeakCan Host v3.8.7-33-gb75317d`. Report a bug → log file says `InformationalVersion=v3.8.7-33-gb75317d`. Support asks "what version?" → user reads the title bar → reports the wrong version.

After: title bar reads `PeakCan Host v3.13.3`. Log file says `InformationalVersion=3.13.3`. User reads the actual installed version from the title bar.

## Lessons (1-of-1)

`window-title-version-must-be-pinned-via-build-properties-not-git-describe-default` — `AssemblyInformationalVersion` defaults to `git describe` output when csproj has no `<Version>` element. On a feature branch with N commits past a tag, the title bar shows `v{oldTag}-N-g{hash}` — unrelated to the PATCH actually shipping. The user can never tell which version they're running. Pattern: every WPF app that displays the version in the title bar MUST pin `<Version>` via `Directory.Build.props` generated by the ship script. The ship script is the single point of truth for "which version is this build".
